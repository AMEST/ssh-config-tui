# Дизайн-документ: консольный менеджер SSH‑конфигураций (TUI)

## 1. Общие сведения

**Проект:** `ssh-config-tui`  
**Платформа:** .NET 8.0, кроссплатформенное приложение (macOS, Linux)  
**Библиотека TUI:** [Terminal.Gui](https://github.com/gui-cs/Terminal.Gui) v2.x  
**Язык:** C# 11+  
**Целевой файл конфигурации:** `~/.ssh/config` (основной) – **все данные, включая пользовательские группы, хранятся в нём же** с помощью специальных комментариев.  
**Лицензия:** MIT (предположительно)

## 2. Цели и задачи

Создать простой, быстрый и интуитивно понятный TUI‑инструмент для повседневного управления SSH‑конфигурациями. Приложение должно позволять:

- Просматривать, редактировать и организовывать SSH‑хосты без прямого вмешательства в текстовый файл.
- Группировать хосты по проектам, окружениям или любым пользовательским категориям, **сохраняя группы прямо в комментариях файла `~/.ssh/config`**.
- Управлять как общими (`Host *`), так и персональными параметрами хостов.
- Сохранять обратно в `~/.ssh/config` с минимальными изменениями форматирования (сохраняя комментарии, пустые строки и порядок директив).
- Работать абсолютно прозрачно: все основные параметры и метаданные находятся в стандартном SSH‑конфиге, понятном для CLI‑инструментов и OpenSSH.

## 3. Функциональные требования

### 3.1. Работа с файлом конфигурации SSH
- **Чтение:** Парсинг `~/.ssh/config` в структурированное внутреннее представление (секции, параметры, комментарии).
- **Запись:** Генерация файла с сохранением исходного форматирования (порядок секций, отступы, комментарии) для нередактированных частей.
- **Поддержка директив:** Host, Match, Hostname, User, Port, IdentityFile, ProxyJump, ForwardAgent, LocalForward, RemoteForward, UserKnownHostsFile, Include и т.д. (полный набор поддерживаемых OpenSSH опций).
- **Include:** Чтение дополнительных файлов, указанных через директиву `Include`. Редактирование возможно только в основном файле (изменять подключаемые файлы через UI не разрешается, но отображать их содержимое можно).
- **Валидация:** Проверка корректности имён хостов, портов, путей к ключам.

### 3.2. Группировка хостов (на базе комментариев)
- **Формат хранения:** для каждого хоста, принадлежащего к одной или нескольким группам, внутри его секции `Host` добавляется строка-комментарий вида `# tui-group: ИмяГруппы1, ИмяГруппы2`. Пример:
  ```
  Host app-server-01
      # tui-group: Production, Critical
      HostName 192.168.1.10
      User root
  ```
  Комментарий размещается сразу после строки `Host ...` (или в начале секции). Всё, что после `# tui-group:` до конца строки, считается списком групп, разделённых запятыми (с учётом пробелов). OpenSSH игнорирует этот комментарий.
- Создание/удаление/переименование групп в рамках приложения автоматически обновляет соответствующие комментарии во всех затронутых хостах.
- При удалении хоста комментарий удаляется вместе с ним.
- Группа «All» (все хосты) формируется динамически из всего списка хостов.
- Группа «Ungrouped» – все хосты, у которых отсутствует строка `# tui-group:`.
- Поддержка переименования группы: поиск всех комментариев с старым именем и замена на новое.

### 3.3. Редактирование хостов
- Добавление нового хоста (с шаблоном или с нуля).
- Редактирование параметров выбранного хоста через форму с полями ввода/выбора.
- Клонирование существующего хоста (включая комментарии групп, если нужно).
- Переименование хоста (с автообновлением ссылок в ProxyJump и т.п. – по желанию).
- Удаление хоста (с подтверждением).

### 3.4. Глобальные параметры
- Редактирование секции `Host *` (значения применяются ко всем хостам).
- Возможность просмотра «эффективных» параметров для конкретного хоста с учётом `Host *`, `Match` и групповых шаблонов.

### 3.5. Дополнительные возможности
- **Поиск / фильтр** по имени хоста, адресу, группе.
- **Проверка SSH-соединения:** запуск `ssh -G <host>` или `ssh -T <host>` для тестирования (результат отображается в отдельном окне).
- **Копирование строки подключения** `ssh <host>` в системный буфер обмена.
- **Сохранение и восстановление сессий:** запоминание открытых вкладок/групп при перезапуске (например, в отдельном dotfile `~/.ssh/config-ui-session.json`, который не влияет на совместимость; это опционально и не хранит метаданные групп – только UI-состояние).
- **Импорт / экспорт** отдельного хоста или группы в формате SSH-конфига (фрагмент).
- **Шаблоны** для быстрого создания хостов с предзаполненными полями (User, Port, IdentityFile и т.д.).

### 3.6. Безопасность
- Проверка прав доступа к `~/.ssh/config` – владелец, разрешения `600`. При несовпадении – предупреждение с предложением исправить.
- Пароли и фразы‑пароли **не сохраняются**; IdentityFile подставляется как путь.
- Никакие ключи, пароли или секреты не попадают в комментарии или отдельные файлы.

## 4. Нефункциональные требования
- **Производительность:** комфортная работа с конфигами до 1000 хостов (отзывчивость интерфейса ≤ 100 мс на типовых операциях).
- **Кроссплатформенность:** macOS (Terminal.app, iTerm2) и Linux (xterm-256color, современные эмуляторы); цветовая схема адаптируется под терминал.
- **Устойчивость:** перед записью создаётся резервная копия (`~/.ssh/config.bak`). Повреждённый конфиг вызывает понятное сообщение об ошибке, а не крах.
- **Расширяемость:** архитектура позволяет в будущем добавить поддержку `Match`‑секций, `Include`‑редактора, SSH‑агента и т.п.

## 5. Архитектура решения

### 5.1. Компонентная модель
Приложение разбивается на следующие слои:

```
ssh-config-tui (Executable)
├── UI (Terminal.Gui Views)
├── Application (Use Cases)
├── Domain (Models)
└── Infrastructure (File I/O, Parsing, Validation)
```

**Краткое описание слоёв:**

- **Domain:**  
  - `SshConfig` – полное представление файла: список узлов (`ConfigNode`).  
  - `ConfigNode` – одна секция (может быть `Host`, `Match` или комментарий / пустая строка).  
  - `HostEntry` – конкретный хост, собирающий параметры из родительских секций, и хранящий список групп (распарсенный из комментария).  
  - `Group` – пользовательская группа (ID = имя, отображаемое имя). Список групп строится динамически на основе комментариев во всех хостах.  
  - `SessionState` (опционально) – состояние UI-сессии (последняя открытая группа, позиция курсора и т.п.), сохраняемое в отдельный dotfile.

- **Infrastructure:**  
  - `SshConfigParser` – парсинг и обратная сериализация с сохранением форматирования. Умеет извлекать комментарии `# tui-group:` и привязывать их к конкретному хосту.  
  - `SshConfigRepository` – чтение/запись `~/.ssh/config`, кэширование, безопасная запись с бекапом, проверка прав.  
  - `SessionStateRepository` – опциональное сохранение/восстановление UI-состояния (не связано с группами).  
  - `SshTester` – сервис для выполнения `ssh -G` / `ssh -T` с обработкой результата.

- **Application:**  
  - `ConfigService` – CRUD хостов и глобальных параметров, разрешение эффективных значений.  
  - `GroupService` – управление группами: получение списка групп из комментариев, добавление/удаление хоста в группу, переименование, синхронизация с `SshConfig`.  
  - `ApplicationService` – координация (загрузка, сохранение, импорт/экспорт).  
  - `ClipboardService` – абстракция над системным буфером обмена (через `Terminal.Gui` или внешние утилиты типа `pbcopy` / `xclip`).

- **UI:**  
  - `MainWindow` – главное окно с MenuBar, StatusBar.  
  - `GroupTreeView` – дерево групп (слева).  
  - `HostListView` – список хостов выбранной группы (справа вверху).  
  - `HostDetailView` – форма редактирования параметров хоста (справа внизу).  
  - `GlobalSettingsView` – модальное окно для `Host *`.  
  - `SearchBar` – верхняя строка для фильтрации.

### 5.2. Взаимодействие компонентов
- Пользователь выбирает группу → `GroupTreeView` вызывает `GroupService.GetHostsByGroup()` → обновляется `HostListView`.  
- Пользователь выбирает хост → `HostDetailView` заполняется через `ConfigService.GetEffectiveConfig(hostName)`.  
- Редактирование полей → `ConfigService.UpdateHost(...)` → при сохранении (авто или ручное) `SshConfigRepository.Save()` записывает файл, предварительно сделав бекап.  
- Любое изменение групп (добавление/удаление хоста из группы) вызывает обновление комментариев `# tui-group:` в соответствующих секциях `Host`.

### 5.3. Модель парсинга конфигурации и хранения групп
Для минимизации диффов при записи используется «исходно‑ориентированный» подход:  
- `SshConfig` хранит **список строк/узлов** в исходном порядке.  
- Каждая не‑комментарийная строка с параметром привязана к конкретному `Host` / `Match` контексту.  
- Специальный комментарий `# tui-group:` распознаётся парсером и сохраняется как свойство узла секции `Host`. При сериализации он вставляется обратно в нужное место (обычно сразу после строки `Host`).  
- При редактировании групп `GroupService` модифицирует эти свойства, а `SshConfigRepository` при сохранении генерирует актуальные комментарии.  
- Неизменённые части файла (включая обычные комментарии и оформление) остаются нетронутыми.

│ 📁 All        │ 🔹 app-server-01   192.168.1.10                  │
│ 📁 Production │ 🔹 db-server-01    10.0.0.5                      │
│ 📁 Staging    │ 🔹 cache-node      10.0.0.6                      │

│               │                                                   │
│               ├───────────────────────────────────────────────────┤

│               │ HostName: 192.168.1.10                            │

```

### 6.2. Основные элементы управления
- **Меню:** `File` (Save, Backup, Import/Export, Quit), `Edit` (Add Host, Clone, Delete, Global Settings), `View` (Groups tree, Refresh), `Help` (About, Keybindings).
- **Группы (левая панель):** дерево, перемещение по стрелкам, `Enter` – выбрать, `Ctrl+N` – новая группа, `Del` – удалить группу (с удалением комментариев из хостов).
- **Список хостов (правая верхняя):** строки с именем хоста и адресом, `j/k` или стрелки, `Enter` – редактировать, `Tab` – переключение на форму деталей.
- **Форма деталей (правая нижняя):** поля для каждого параметра, `Tab` между полями, `Ctrl+S` – сохранить хост, `Esc` – отменить редактирование.
- **Горячие клавиши:**  
  - `F5` – Refresh (перезагрузить конфиг с диска).  
  - `Ctrl+F` – поиск.  
  - `Ctrl+T` – проверить соединение с выбранным хостом.  
  - `Ctrl+C` – скопировать ssh-строку.

### 6.3. Редактор параметров хоста
Отдельное окно / вкладка с динамическим списком полей. Для каждого параметра – текстовое поле или выпадающий список (например, для выбора файла ключа можно показать файловый диалог, упрощённый для терминала). Кнопки «Add Parameter» для добавления нестандартных опций.

### 6.4. Работа с группами
- Создать группу – всплывающее окно с вводом имени (только имя, создаётся виртуально; фактически группа появляется, когда хосту назначается этот тег).
- Переименовать / удалить – контекстное меню (или по `F2`/`Del`).
- Назначение хостов в группы:
  - В списке хостов `Space` – пометить хост, затем контекстное меню «Add to group…» / «Remove from group…».
  - Или выделить хост, нажать `g` и выбрать группу из списка.
- Группы автоматически синхронизируются с комментариями `# tui-group:` при сохранении.

## 7. Поток работы (workflow)

1. **Запуск приложения** → парсинг `~/.ssh/config` (включая извлечение групп из комментариев) → построение модели.
2. Отображение главного экрана с активной последней группой (или «All») – восстановленной из сессионного файла (если реализован).
3. Пользователь выбирает группу → обновляется список хостов.
4. Пользователь выбирает хост → заполняется форма деталей.
5. Изменения в форме **не сохраняются автоматически**, пока пользователь не нажмёт `Save` (или `Ctrl+S`).
6. При сохранении:
   - Валидация полей.
   - Обновление внутренней модели, включая комментарии групп.
   - Сериализация `SshConfig` в буфер (все секции, строки, комментарии).
   - Создание бэкапа `~/.ssh/config.bak`.
   - Запись в `~/.ssh/config` с атомарным переименованием временного файла.
7. При выходе (`Ctrl+Q`) – предупреждение о несохранённых изменениях.

## 8. Обработка ошибок и граничные случаи
- **Файл конфига не существует:** создаётся пустой файл с правами 600.
- **Некорректный синтаксис:** отображается сообщение с номером строки, предлагается открыть в редакторе (или внутренний просмотрщик).
- **Одновременное редактирование:** блокировка не реализуется, но перед записью проверяется время изменения файла и, если оно расходится, выдаётся предупреждение о перезаписи.
- **Длинные файлы (1000+ хостов):** используется виртуализация в `ListView` через адаптер данных (Terminal.Gui поддерживает это).
- **Специальные символы в именах/значениях:** проброс через SSH-парсер в точности, как этого ожидает OpenSSH.
- **Повреждённый комментарий группы:** если строка `# tui-group:` содержит невалидные символы, она игнорируется, но сохраняется в файле «как есть» (чтобы не удалять пользовательские правки). Приложение показывает предупреждение при загрузке.

## 9. Тестирование
- **Модульное тестирование:** парсера конфигов (включая извлечение/изменение комментариев групп), сервисов CRUD, модели эффективных параметров.
- **Интеграционное тестирование:** реальный `ssh -G` с тестовыми конфигами (можно на CI без реальных серверов).
- **Тестирование UI:** скриншот-тесты через `Terminal.Gui` testing framework (если доступен), либо эмуляция ввода через консольные тесты.
- **Платформенное тестирование:** ручные прогоны на macOS (zsh), Ubuntu (bash), Fedora.

## 10. План разработки (вехи)
1. **PoC:** парсер/сериализатор `~/.ssh/config` с поддержкой комментариев и извлечением групп, базовая модель данных.
2. **Первый рабочий TUI:** просмотр хостов без групп, редактирование одного хоста, сохранение (с сохранением комментариев).
3. **Управление группами:** отображение групп на основе комментариев, UI дерева групп, назначение/снятие группы у хоста, синхронизация комментариев при сохранении.
4. **Расширенное редактирование:** все основные параметры, добавление/удаление опций, клонирование, глобальные настройки.
5. **Дополнительные фичи:** поиск, тестирование соединения, копирование в буфер, шаблоны, импорт/экспорт.
6. **Полировка и стабилизация:** обработка ошибок, бэкап, производительность, сессионное состояние.

## 13. Пошаговая инструкция для реализации (джуниору или AI-агенту)

Ниже приведён детальный roadmap, разбитый на конкретные задачи. Каждый шаг может быть поручен отдельному исполнителю или агенту.

### Шаг 0. Подготовка проекта
- Создать новый .NET консольный проект: `dotnet new console -n ssh-config-tui`.
- Добавить NuGet-пакеты: `Terminal.Gui`, `Microsoft.Extensions.DependencyInjection`, `CliWrap` (для запуска внешних команд), тестовые фреймворки.
- Настроить DI-контейнер в `Program.cs`.
- Создать структуру каталогов: `Domain/`, `Infrastructure/`, `Application/`, `UI/`.

### Шаг 1. Парсер и модель SSH-конфига (Infrastructure + Domain)
- **Задача:** разработать класс `SshConfigParser`, который читает `~/.ssh/config` и возвращает объект `SshConfig`.
- `SshConfig` – коллекция узлов (`ConfigNode`). Каждый узел может быть:
  - `HostSection` (имя хоста, список параметров, список комментариев, включая специальный `tui-group`).
  - `MatchSection` (пока можно только читать как есть, без редактора).
  - `CommentLine` / `EmptyLine` – для сохранения форматирования.
- Парсер должен разбирать строки по пробелам, различать ключ-значение параметров, обрабатывать многострочные значения (кавычки, экранирование).
- **Специальный комментарий:** `# tui-group: ...` должен распознаваться и сохраняться в свойстве `Groups` у `HostSection` (список строк). При сериализации обратно в текст комментарий вставляется сразу после строки `Host ...`.
- Написать юнит-тесты: корректный парсинг простого конфига, конфига с группами, сохранение неизменённых комментариев и порядка строк, обработка синтаксических ошибок.

### Шаг 2. Репозиторий и базовые операции (Infrastructure)
- `SshConfigRepository`:
  - `LoadAsync()`: читает файл, парсит, кэширует.
  - `SaveAsync(SshConfig config)`: создаёт бэкап `~/.ssh/config.bak`, записывает новый конфиг с атомарным переименованием временного файла (чтобы избежать повреждения при сбое).
  - `ValidatePermissions()`: проверяет, что файл имеет права 600 и принадлежит текущему пользователю. Если нет – выбрасывает предупреждение (но не блокирует работу).
- Добавить DI-регистрацию репозитория как singleton.

### Шаг 3. Сервис приложения и основные use cases (Application)
- `ConfigService`:
  - Методы: `GetAllHosts()`, `GetHost(string name)`, `AddHost(...)`, `UpdateHost(...)`, `DeleteHost(name)`, `GetGlobalConfig()`.
  - `GetEffectiveConfig(hostName)` – агрегирует параметры от `Host *` и конкретного хоста (пока без `Match`).
- `GroupService`:
  - `GetGroups()`: сканирует все `HostSection` в `SshConfig`, собирает уникальные имена групп из комментариев `# tui-group:`.
  - `GetHostsByGroup(string group)`: фильтрует хосты по наличию группы.
  - `AddHostToGroup(hostName, group)`: добавляет группу в комментарий хоста.
  - `RemoveHostFromGroup(...)`: удаляет.
  - `RenameGroup(oldName, newName)`: проходит по всем хостам, заменяет имя в их комментариях.

## 14. Автоматизация сборки и публикации через GitHub Actions

Релиз запускается при публикации тега (например, `v1.0.0`). Workflow собирает:

- **Self-contained бинарники** для macOS (osx-x64, osx-arm64) и Linux (linux-x64, linux-arm64). Каждый бинарник упаковывается в `.tar.gz` и прикрепляется к релизу.
- **NuGet пакет dotnet tool** публикуется в NuGet.org (или во встроенный GitHub Packages), чтобы можно было установить командой `dotnet tool install -g ssh-config-tui`.

### 14.1. Workflow файл (`.github/workflows/release.yml`)

```yaml
name: Release

on:
  push:
    tags: [ 'v*' ]

env:
  DOTNET_VERSION: '8.0.x'
  PROJECT_PATH: src/ssh-config-tui/ssh-config-tui.csproj

jobs:
  build-and-release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest]
        include:
          - os: ubuntu-latest
            rid: linux-x64
            artifact_name: ssh-config-tui-linux-x64
          - os: ubuntu-latest
            rid: linux-arm64
            artifact_name: ssh-config-tui-linux-arm64
          - os: macos-latest
            rid: osx-x64
            artifact_name: ssh-config-tui-osx-x64
          - os: macos-latest
            rid: osx-arm64
            artifact_name: ssh-config-tui-osx-arm64

    steps:
    - uses: actions/checkout@v4

    - name: Setup .NET
      uses: actions/setup-dotnet@v4
      with:
        dotnet-version: ${{ env.DOTNET_VERSION }}

    - name: Publish self-contained
      run: |
        dotnet publish ${{ env.PROJECT_PATH }} \
          -c Release \
          -r ${{ matrix.rid }} \
          --self-contained true \
          -p:PublishSingleFile=true \
          -p:PublishTrimmed=true \
          -o publish/${{ matrix.artifact_name }}
        tar -czf ${{ matrix.artifact_name }}.tar.gz -C publish/${{ matrix.artifact_name }} .

    - name: Upload artifact
      uses: actions/upload-artifact@v4
      with:
        name: ${{ matrix.artifact_name }}
        path: ${{ matrix.artifact_name }}.tar.gz

  publish-nuget:
    needs: build-and-release
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Setup .NET
      uses: actions/setup-dotnet@v4
      with:
        dotnet-version: ${{ env.DOTNET_VERSION }}
    - name: Pack dotnet tool
      run: dotnet pack ${{ env.PROJECT_PATH }} -c Release -o nuget
    - name: Push to NuGet.org
      run: dotnet nuget push nuget/*.nupkg --api-key ${{ secrets.NUGET_API_KEY }} --source https://api.nuget.org/v3/index.json

  create-release:
    needs: build-and-release
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
    - name: Download all artifacts
      uses: actions/download-artifact@v4
    - name: Create GitHub Release
      uses: softprops/action-gh-release@v1
      with:
        files: '**/*.tar.gz'
        generate_release_notes: true
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Примечания:**
- `PublishTrimmed` можно включить для уменьшения размера, но следует проверить совместимость с Terminal.Gui.
- Для публикации в NuGet.org необходим секрет `NUGET_API_KEY`.
- Сборка для ARM64 Linux может потребовать дополнительной настройки кросс-компиляции; при проблемах можно оставить только linux-x64 и оба варианта macOS.
- Подпись кода отсутствует (дорого), поэтому для macOS потребуется инструкция по обходу Gatekeeper (см. README).

### Running on macOS

Since the binary is not signed with an Apple Developer certificate, macOS Gatekeeper will block it on first launch.

After downloading and extracting the `ssh-config-tui` binary:

1. Open Terminal.
2. Navigate to the directory containing the binary.
3. Remove the quarantine attribute:
   ```bash
   xattr -d com.apple.quarantine ./ssh-config-tui
   ```
4. Make it executable:
   ```bash
   chmod +x ./ssh-config-tui
   ```
5. Run it:
   ```bash
   ./ssh-config-tui
   ```

If you still see a warning, go to **System Settings > Privacy & Security** and click "Allow Anyway" next to the message about `ssh-config-tui`.
```
